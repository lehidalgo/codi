# FTS5 reindex runbook

- Date: 2026-05-13 23:00 UTC
- Document: 20260513*230000*[GUIDE]\_fts5-reindex.md
- Category: GUIDE

## When to run

The brain DB carries two FTS5 virtual tables that index `captures.content`
and `prompts.text`:

- `captures_fts`
- `prompts_fts`

Triggers in `src/runtime/brain/migrate.ts` keep both indexes in sync on
every INSERT / UPDATE / DELETE / soft-delete. A rebuild is needed only
when one of the following happens:

1. Manual edits to `captures.content` or `prompts.text` outside the
   normal capture pipeline (rare — usually only during debugging).
2. The trigger block was disabled at some point and restored later.
3. The FTS5 internal store is corrupted (rare — surfaces as the search
   query returning rows whose `capture_id` no longer exists in
   `captures`, or `SELECT * FROM captures_fts WHERE captures_fts MATCH
'*'` throwing "vtable constructor failed").

If you do not see a real symptom, do not run this. The triggers are the
authoritative path.

## What it does

Drops the FTS5 internal shadow tables and rebuilds them from the
parent table. Both `captures_fts` and `prompts_fts` use the
contentless-rowid pattern, so the rebuild reads `captures.content` and
`prompts.text` directly and refills the index.

The rebuild holds a write lock on the parent tables for the duration of
the operation. On a 100k-row brain it takes ~5-30 seconds.

## Procedure

1. Stop any running brain-ui server (it holds a connection but a
   reindex tolerates concurrent readers; stopping is defensive).

   ```bash
   codi brain ui --background false   # if currently running attached
   pkill -f brain-ui-server.js         # if running detached
   ```

2. Run the reindex SQL against the brain. Replace `<brain>` with the
   actual path (default `~/.codi/state/brain.db`).

   ```bash
   sqlite3 "$HOME/.codi/state/brain.db" <<'SQL'
   INSERT INTO captures_fts(captures_fts) VALUES('rebuild');
   INSERT INTO prompts_fts(prompts_fts)   VALUES('rebuild');
   SQL
   ```

3. Verify the indexes are non-empty and the count matches the parent
   tables.

   ```bash
   sqlite3 "$HOME/.codi/state/brain.db" <<'SQL'
   SELECT
     (SELECT COUNT(*) FROM captures WHERE deleted_at IS NULL) AS captures_alive,
     (SELECT COUNT(*) FROM captures_fts)                       AS captures_fts_rows,
     (SELECT COUNT(*) FROM prompts)                            AS prompts_total,
     (SELECT COUNT(*) FROM prompts_fts)                        AS prompts_fts_rows;
   SQL
   ```

   The first row of the result should show:
   - `captures_fts_rows >= captures_alive` (soft-deleted captures remain
     indexed until the soft-delete trigger fires `INSERT INTO
captures_fts(captures_fts, ...) VALUES('delete', ...)`).
   - `prompts_fts_rows == prompts_total` (prompts are never soft-deleted).

4. Quick sanity match — pick a known capture word and confirm the
   search route returns it.

   ```bash
   sqlite3 "$HOME/.codi/state/brain.db" <<'SQL'
   SELECT capture_id, substr(content, 1, 60)
   FROM captures_fts
   WHERE captures_fts MATCH 'codi'
   LIMIT 5;
   SQL
   ```

5. Restart brain-ui if you stopped it.

   ```bash
   codi brain ui
   ```

## Failure recovery

If `INSERT … VALUES('rebuild')` fails with "no such module: fts5":

- The brain was opened by a `better-sqlite3` build that lacks the
  FTS5 amalgamation. This should not happen in production — `pnpm
rebuild better-sqlite3` recompiles with FTS5 enabled per the prebuilt
  build matrix.

If the rebuild succeeds but `MATCH` queries still miss rows:

- Confirm the parent rows actually exist (run `SELECT COUNT(*) FROM
captures` — should be > 0).
- Confirm `deleted_at IS NULL` for the row you expected — soft-deleted
  rows are intentionally not searchable.
