# Content Factory Design System

Core design rules for carousels, slide decks, and social cards generated by
content-factory. These rules are **not optional** — they produce consistent,
readable, on-brand output across every slide in a deck.

Apply them as you write HTML/CSS. When an existing rule conflicts with a new
request, the rule wins unless the user explicitly overrides it.

---

## 1. Fixed frame: chrome + header + content zone + chrome-bottom

Every slide in a carousel has **four zones stacked vertically**, and three of
them must be **identical in position and size** across every slide:

```
┌─ slide canvas (e.g. 1080 × 1350) ──────────────────────┐
│                                                        │
│   [ chrome-top ]         ← FIXED position & height     │
│                                                        │
│   ─────────────────────────────────                    │
│   h2 title                                             │
│   secondary description (.sub / .feat-lead)            │
│   ─────────────────────────────────                    │
│                                                        │
│   ╔═══════════════════════════╗                        │
│   ║                           ║                        │
│   ║     CONTENT ZONE          ║   ← FIXED zone,        │
│   ║     (fills remaining      ║     content must fit   │
│   ║      vertical space)      ║                        │
│   ║                           ║                        │
│   ╚═══════════════════════════╝                        │
│                                                        │
│   [ chrome-bottom ]      ← FIXED position & height     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**Chrome-top**: brand wordmark left, slide counter + feature counter right.
Always at the top edge, always the same height.

**Header** (h2 + secondary description): title + one-line lead. Locked height
via `min-height` on h2 and the secondary description.

**Content zone**: the only zone that varies per slide. All feature content
lives here. Must *always* fit — never overflow. If it doesn't fit, compact
the content, do not expand the zone.

**Chrome-bottom**: handle, repo link, call-to-action. Always at the bottom
edge, always the same height.

### How to enforce it

```css
/* Header elements reserve fixed vertical space */
h2                      { min-height: 120px; }   /* ~2 lines at 60px */
.sub, .feat-lead        { min-height: 80px;  }   /* ~2 lines at 28px */

/* Body flows as flex column with uniform gap */
.body { display: flex; flex-direction: column; flex: 1; gap: 20px; }
```

### Verification

After any edit, measure these positions across slides 2–N (cover often
differs). They must match (±1px rounding):

```js
[...document.querySelectorAll('.social-card')].map(c => {
  const ct = c.querySelector('.chrome-top').getBoundingClientRect();
  const cb = c.querySelector('.chrome-bottom').getBoundingClientRect();
  return {
    chromeTopBottom: Math.round(ct.bottom),
    chromeBottomTop: Math.round(cb.top),
  };
});
```

## 2. Chrome indicators must match

Slide chrome carries up to two pill-shaped indicators in the top-right:

- **Slide counter** (`.slide-num` → `05 / 10`)
- **Feature counter** (`.chrome-feat` → `FEATURE 2 / 5`) — optional, only on
  feature slides

**Both pills must share identical styling.** The user will immediately notice
any difference in font-size, padding, border-radius, or color.

```css
.slide-num, .chrome-feat {
  font-family: var(--ff-m);
  font-size: 18px;
  color: var(--text-3);
  letter-spacing: 0.1em;
  text-transform: uppercase;
  font-weight: 500;
  padding: 6px 11px;
  border: 1px solid var(--border);
  border-radius: 5px;
  background: rgba(0,0,0,0.3);
}
.slide-num b, .chrome-feat b {
  color: var(--c0);
  font-weight: 700;
  font-size: 18px;        /* same size as label — only weight and color change */
  margin-left: 10px;
  letter-spacing: 0.1em;
}
```

Notes:
- **Never** use two different CSS classes with slightly different sizes — one
  canonical rule, two class selectors.
- The cyan `<b>` number must be the same font-size as the label. Differentiate
  via color and weight, not size.
- Pill height must be identical when measured (both render at ~38px with 18px
  font and `6px 11px` padding).

### Verification

```js
const card = document.querySelectorAll('.social-card')[5];
const cf = card.querySelector('.chrome-feat').getBoundingClientRect();
const sn = card.querySelector('.slide-num').getBoundingClientRect();
console.log({ cfH: cf.height, snH: sn.height });   // must be equal
```

## 3. Text size standard

All **plain body text** sits within this 5-tier hierarchy:

| Tier | Size | Role | Example |
|------|------|------|---------|
| T1 | **28px** | Secondary description below title | `.sub`, `.feat-lead` |
| T2 | **26px** | Primary body text inside content cards | pain description, cover hook, summary body |
| T3 | **24px** | Supporting body text | stat descriptions, bullet items, step labels |
| T4 | **22px** | Tertiary body text | flow notes, stat-card captions |
| T5 | **20px** | **Minimum** — small descriptions only | quote spans, chrome handle, captions |

**Rule**: no body text smaller than 20px. Readers on phones and in social
feeds can't reliably read smaller.

### Allowed exceptions (non-text)

These can sit **outside** the 20–28px range because they are not plain text:

- **Titles and headings** (h1, h2, h3) — typically 30–96px
- **Big numerals** — `.num`, `.val`, decorative gradient digits (46–78px)
- **Chrome counters** — slide numbers, feature counters (18px — see rule 2)
- **Pill labels and badges** — category tags, status badges (11–15px)
- **Monospace code blocks and terminals** — (18–22px — monospace reads
  slightly narrower than sans-serif at the same px)
- **Decorative logo wordmarks** — (40–96px)

### How to apply

Whenever you write a new text element, map it to a tier:

```css
.card-description   { font-size: 26px; }  /* T2 — primary body */
.card-caption       { font-size: 22px; }  /* T4 — tertiary */
.card-metadata      { font-size: 20px; }  /* T5 — minimum */
```

Do **not** use arbitrary sizes like 19px, 21px, 23px, 27px. Stick to the 5
tiers.

## 4. Two-color title pattern (white + gradient accent)

Every h2 and h3 title in a deck should split into two color zones:

- **Plain white text** for the setup / subject
- **Cyan gradient** (`<span class="grad">`) for the punch, consequence, or
  value phrase

```html
<h2>Rules, skills, agents —<br><span class="grad">one lifecycle.</span></h2>
<h3>Standards rot <span class="grad">in silence</span></h3>
```

The gradient lands on the **most impactful word or phrase** in the title. If
you're not sure which half to gradient, pick the one that completes the
emotional beat — usually the second half.

```css
h2 .grad, h3 .grad {
  background: var(--grad);
  -webkit-background-clip: text;
  -webkit-text-fill-color: transparent;
}
```

**Never** put both halves of a title in the same color — the two-tone pattern
is the signature of the deck.

## 5. Equidistant spacing rule

Inside any container, every direct child must sit at an **equal distance**
from its siblings.

### What this means

- **Use `gap` on the container**, not `margin-top`/`margin-bottom` on children
- **Do not** add one-off margins to individual siblings to nudge spacing
- **Do not** use negative margins as corrections
- **Container padding** is symmetric unless asymmetry is intentional and
  justified (e.g., a reserved absolute-positioned element in the top-left)
- The rule is **recursive** — it applies to the body container, and to every
  nested card, and to every row inside those cards

### Before (violates the rule)

```css
.body { display: flex; flex-direction: column; }
.body h2               { margin-bottom: 18px; }
.body .sub             { margin-bottom: 16px; }
.body .feat-stats      { margin-bottom: 14px; }
.body .feat-bullets    { margin-bottom: 12px; }
.body .feat-terminal   { /* no margin */ }
```

Each gap is a different value. Rhythm is broken.

### After (respects the rule)

```css
.body {
  display: flex;
  flex-direction: column;
  gap: 20px;    /* single source of truth */
}
.body h2, .body .sub, .body .feat-lead,
.body .feat-stats, .body .feat-bullets, .body .feat-terminal {
  margin: 0;    /* no per-child overrides */
}
```

All pairs of adjacent siblings are exactly 20px apart.

### Inline style overrides are forbidden

```html
<!-- BAD -->
<div class="mess-grid" style="margin-top:20px;">

<!-- GOOD -->
<div class="mess-grid"><!-- spacing comes from parent .body { gap } --></div>
```

## 6. Three-element pinned card layout (lbl / val / note)

The canonical feature-slide stat card has exactly three elements. They must
line up horizontally across every card in the row regardless of how much text
each one holds.

```
┌─ .feat-stat ─────────────────────────┐
│                                      │
│  ● LBL               ← position:abs  │   anchored at top:26px, left:28px
│                                      │
│                                      │
│        VAL           ← flex:center   │   flex-centered in the padded area
│                                      │
│                                      │
│  ────────────────                    │   border-top on .note
│  NOTE NOTE NOTE      ← position:abs  │   anchored at bottom:26px with
│  NOTE NOTE NOTE                      │   fixed height (e.g. 115px)
│                                      │
└──────────────────────────────────────┘
```

### Why this layout

- **lbl** (small accent label, e.g. `● RULES`): `position: absolute` at the
  top-left. Every card's lbl renders at the **exact same pixel offset**
  regardless of card height.
- **val** (the headline number or phrase, e.g. `27+`, `state.json`,
  `codi doctor`): the **only** flex child of the card's flex column, centered
  with `justify-content: center`. The val's visual center sits at the card's
  visual center line — consistent across cards of equal height.
- **note** (caption under a separator line, e.g.
  `coding standards your agent always loads`): `position: absolute` at the
  bottom, with a **fixed height** so the separator line (border-top) renders
  at the same y-position on every card, even when notes have different line
  counts.

### Canonical CSS

```css
.feat-stats {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 12px;
  align-items: stretch;           /* grid stretches cells to row height */
  flex: 1;                         /* the stats grid fills body space */
}

.feat-stat {
  background: var(--surface-2);
  border: 1px solid var(--border);
  border-radius: 14px;
  padding: 64px 28px 155px;       /* top: lbl zone; bottom: note zone */
  position: relative;
  overflow: hidden;
}

/* Flex container for the centered val (only flow child) */
.feat-stats .feat-stat {
  display: flex;
  flex-direction: column;
  justify-content: center;
  gap: 0;
}

/* Element 1: label, pinned top */
.feat-stat .lbl {
  position: absolute;
  top: 26px;
  left: 28px;
  font-size: 15px;
  color: var(--c0);
  letter-spacing: 0.18em;
  text-transform: uppercase;
  font-weight: 700;
}

/* Element 2: val, flex-centered */
.feat-stat .val {
  font-size: 52px;
  color: var(--text-1);
  font-weight: 800;
  line-height: 1.05;
  letter-spacing: -0.025em;
  font-family: var(--ff-h);
  margin: 0;
}
.feat-stat .val.g {
  background: var(--grad);
  -webkit-background-clip: text;
  -webkit-text-fill-color: transparent;
}

/* Element 3: note, pinned bottom with fixed height */
.feat-stat .note {
  position: absolute;
  bottom: 26px;
  left: 28px;
  right: 28px;
  height: 115px;                  /* reserve ~3 lines at T4 + separator */
  font-size: 22px;                /* T4 */
  font-family: var(--ff-m);
  color: rgba(139,148,158,0.85);
  line-height: 1.45;
  font-weight: 400;
  border-top: 1px solid rgba(86,182,194,0.18);
  padding-top: 12px;
}
```

### The 155px bottom padding

The `.feat-stat` bottom padding must account for the note zone. It equals
`note.height + note.bottom + ~15px safety margin` so the flex-centered val
never collides with the top of the note:

```
155px ≈ 115 (note height) + 26 (note bottom offset) + 14 (safety breathing)
```

If you change the note height, update the card bottom padding in the same
edit.

### Multi-line val values

If vals can span multiple lines (e.g. `off<br>warn<br>err`), the grid row
stretches to accommodate the tallest card — `align-items: stretch` keeps the
rest of the row equal. The flex-centering places each val at its own visual
center, and the lbl + note remain aligned across cards because they are
absolutely positioned.

### Verification

```js
const card = document.querySelectorAll('.social-card')[4];  // slide 5
const rect = card.getBoundingClientRect();
const lbls = [...card.querySelectorAll('.feat-stat .lbl')].map(e => Math.round(e.getBoundingClientRect().top - rect.top));
const notes = [...card.querySelectorAll('.feat-stat .note')].map(e => Math.round(e.getBoundingClientRect().top - rect.top));
console.log({ lbls, notes });
// lbls: all equal, e.g. [374, 374, 374]
// notes: all equal, e.g. [472, 472, 472]
```

## 7. Feature slide layout hierarchy (stats > bullets > terminal)

Feature slides combine up to three body components after the header:

1. **stat grid** (`.feat-stats`) — required
2. **bullet card** (`.feat-bullets`) — optional
3. **terminal card** (`.feat-terminal`) — optional

The **stat grid gets the most visual real estate**. Bullets and terminal are
**content-sized** so they don't steal space from the stats.

```css
.feat-stats    { flex: 1;       }   /* grows to fill remaining body space */
.feat-bullets  { flex: 0 0 auto; }   /* sized exactly to content */
.feat-terminal { flex: 0 0 auto; }   /* sized exactly to content */
```

### Why

- Stat cards carry the hero numbers — they are the primary value proposition
  of the slide and need visual weight.
- Bullets and terminals are **supporting** content. Giving them `flex: 1`
  makes them stretch vertically, which creates awkward empty space between
  their items.
- With `flex: 0 0 auto`, their height = the natural height of their content,
  so there is never dead vertical space inside them.

## 8. Bullet uniform spacing

Inside a `.feat-bullets` card, the space above the first item, between
items, and below the last item must all be equal. Use **`space-evenly`**
combined with **zero vertical container padding** and **zero vertical item
padding**:

```css
.feat-bullets {
  background: var(--surface-2);
  border: 1px solid var(--border);
  border-radius: 14px;
  padding: 0 26px;                /* zero vertical padding */
  list-style: none;
  flex: 0 0 auto;
  display: flex;
  flex-direction: column;
  justify-content: space-evenly;  /* top/between/bottom all equal */
}
.feat-bullets li {
  font-size: 24px;                /* T3 */
  line-height: 1.35;
  color: var(--text-2);
  padding: 14px 0 14px 44px;      /* uniform top/bottom padding + arrow indent */
  position: relative;
  border-bottom: 1px solid var(--border);
}
.feat-bullets li:last-child { border-bottom: none; }
```

**Never** mix per-item vertical padding with `gap` or `justify-content:
space-around` — those approaches create uneven edge spacing that the eye
picks up immediately.

## 9. Content area overflow policy

**The content area cannot overflow**. Ever. After every structural edit:

```js
[...document.querySelectorAll('.social-card')].map((c, i) => {
  const cr = c.getBoundingClientRect();
  const over = [...c.querySelectorAll('*')].filter(el => {
    const r = el.getBoundingClientRect();
    return r.bottom > cr.bottom + 2 || r.right > cr.right + 2;
  }).length;
  return { slide: i+1, overflow: over };
});
```

All slides must return `overflow: 0`. If any slide overflows:

1. **Compact the content** (reduce padding, trim text, shrink non-text
   elements like terminals or code blocks)
2. **Do not** expand the canvas
3. **Do not** shrink body text below the 20px minimum (rule 3)
4. **Do not** remove the fixed frame zones (rule 1)

## 10. Quote / dialogue box pattern

When quoting user dialogue, PR comments, or overheard snippets inside a
content card:

```css
.quote {
  background: var(--surface-3);
  border: 1px dashed var(--border-a);
  border-radius: 10px;
  padding: 16px 22px;
  font-family: var(--ff-m);
  position: relative;
}
.quote span {
  font-size: 20px;             /* T5 — minimum */
  font-style: italic;
  color: var(--text-1);
  padding-left: 24px;          /* reserve space for decorative " */
}
.quote span::before {
  content: "\201C";            /* big cyan " mark */
  position: absolute;
  left: 20px;
  top: -8px;
  font-size: 34px;
  color: var(--c0);
  opacity: 0.55;
}
.quote::after {
  content: "";                 /* glowing status dot top-right */
  position: absolute;
  top: 16px; right: 20px;
  width: 10px; height: 10px;
  border-radius: 50%;
  background: var(--c0);
  box-shadow: 0 0 10px var(--c0);
}
```

## 11. Chrome-bottom handle

The author handle in `.chrome-bottom` renders without a decorative dot. If a
previous template included a `::before` dot, explicitly disable it:

```css
.chrome-bottom .handle::before { display: none; }
```

This keeps the bottom chrome clean and the handle reads as a single text
unit.

## 12. Verification checklist — before calling a slide done

Run this mental checklist on every slide you edit:

1. **Frame locked** — chrome-top, header (h2 + lead), chrome-bottom at the
   same y-positions as every other slide
2. **Chrome indicators match** — `.slide-num` and `.chrome-feat` render at the
   same pill height with the same font/padding/border
3. **Text in tier** — every plain-text element maps to T1–T5; nothing smaller
   than 20px and nothing between tiers (no 19, 21, 23, 27)
4. **Two-color titles** — every h2/h3 uses `<span class="grad">` on the
   punch phrase
5. **Equidistant** — `.body` uses a single `gap`; no per-child margins; no
   inline `style="margin-top"` overrides
6. **Stat cards pinned** — lbl absolute top, val flex-centered, note absolute
   bottom with fixed height; lbl and note y-positions match across all cards
   in the row
7. **Feature hierarchy** — stats `flex: 1`, bullets and terminal
   `flex: 0 0 auto`
8. **Bullet uniform spacing** — `space-evenly` with zero container vertical
   padding
9. **No overflow** — run the verification script; all slides return 0
10. **Equidistant recursive** — every nested card also uses `gap`, no inner
    `margin-top`/`margin-bottom` shortcuts

If any item fails, fix before continuing. Do not ship a slide that violates
the design system.

## 13. Cover slide exception

The **cover slide** (first slide of a carousel) is allowed to deviate from
rule 1 (fixed frame). It commonly uses:

- A large brand wordmark in the chrome-top slot (bigger than the logo on
  other slides)
- An `h1` instead of an `h2`, at a larger size (80–96px)
- A hero visual (logo orbit, big headline-only composition, etc.) instead of
  the standard header + content zone split

The cover is the scroll-stopper — it breaks the grid on purpose to grab
attention. Rules 3 (text size), 5 (equidistance), and 9 (no overflow) still
apply.

## Origin

This system was distilled from the Codi LinkedIn launch carousel design work
(April 2026). Every rule here was derived from a specific visual defect

margin.
