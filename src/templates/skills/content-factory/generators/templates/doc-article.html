codi The Problem
The Problem

Config
sprawl

Every AI coding tool ships with its own config format. Claude Code reads CLAUDE.md, Cursor reads .cursorrules, Copilot reads its own instructions file, Windsurf reads .windsurfrules.

Teams end up manually maintaining 5 separate config files with duplicate rules, diverging conventions, and no single source of truth. Update one — the others fall behind.

The compounding cost

The deeper problem is not just duplication — it is drift. When a new security rule is added to your Claude config, does Cursor know about it? When you onboard a new teammate using Copilot, do they get the same context as everyone else?

Config sprawl turns every rule change into a multi-file operation. It turns onboarding into archaeology. It makes "what does our team AI actually know?" an unanswerable question.

codi Get Started
Get Started

Ship
today

Getting started takes two commands:

$npm install -g codi-cli
$codi init

codi detects your project stack, generates starter rules, and sets up config for every AI tool you use. From there, edit rules in .codi/rules/ and run codi generate to propagate changes everywhere.

What you get

A single .codi/ directory becomes the source of truth for all your AI agent configuration. Rules, skills, agents, and hooks — all in one place, all versioned, all shareable.

Open source. codi is MIT licensed. Contribute rules and skills back to the community via codi contribute.
