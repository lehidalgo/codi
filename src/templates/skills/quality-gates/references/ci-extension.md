# CI workflow extension

The `setup` mode can extend `.github/workflows/*.yml` to mirror local hooks at the authoritative gate. CI changes are HARD-GATED — explicit user approval per file, per change.

## Why mirror locally + CI

Catch failures in seconds locally instead of after a 2-minute CI round trip. Same checks at both gates means a passing local push cannot fail CI on hook-shaped issues. CI remains authoritative for environment-specific failures (OS, Node/Python version drift, dependency resolution differences).

## What the extension adds

The skill proposes adding (or merging into) the existing CI workflow:

```yaml
jobs:
  quality-gates:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0 # branch-naming validator needs full history

      - name: Setup pnpm (if JS/TS)
        if: hashFiles('package.json') != ''
        uses: pnpm/action-setup@v4
        with: { version: 10 }

      - name: Setup Node (if JS/TS)
        if: hashFiles('package.json') != ''
        uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }

      - name: Setup Python (if Python)
        if: hashFiles('pyproject.toml') != ''
        uses: actions/setup-python@v5
        with: { python-version: "3.12" }

      - name: Install dependencies
        run: |
          if [ -f pnpm-lock.yaml ]; then pnpm install --frozen-lockfile; fi
          if [ -f pyproject.toml ]; then pip install pre-commit; fi

      - name: Run gitleaks
        uses: gitleaks/gitleaks-action@v2

      - name: Branch-naming validator
        run: bash skills/quality-gates/scripts/verify-branch-name.sh

      - name: Run pre-commit hooks (full repo)
        run: |
          if [ -f .pre-commit-config.yaml ]; then
            pre-commit run --all-files
          else
            # husky path — emulate pre-commit by running each hook command on all files
            bash skills/quality-gates/scripts/run-husky-on-all.sh
          fi

      - name: Type-check + tests + coverage
        run: |
          if [ -f package.json ]; then pnpm run validate; fi
          if [ -f pyproject.toml ]; then uv run pytest --cov; fi
```

## HARD GATE flow

When the user invokes `quality-gates setup`:

1. Detect existing `.github/workflows/*.yml` files.
2. Compute the diff between current CI and the proposed addition.
3. Display the diff to the user with a short summary ("Adds quality-gates job; takes ~30s; mirrors local pre-commit + pre-push").
4. Ask for explicit approval (recommended answer: yes for fresh repos, "show me the diff once more" for active CI).
5. Only on approval, write the changes.
6. Record the approval in `git config codi.qualityGates.ciApproved <commit-sha>` so `setup` can detect the prior approval and skip re-asking on idempotent re-runs (unless the diff has changed).

## Anti-patterns

- Auto-modifying CI without approval. The CI is shared infrastructure; one bad workflow change fails every PR.
- Replacing the existing CI workflow wholesale. The skill MERGES — never overwrites.
- Adding the quality-gates job before the existing test job. Existing test must remain authoritative.
- Skipping the diff display when the user is in a hurry — the diff is the contract.

## Forbidden-trailers check (separate from quality-gates)

The CI workflow also runs a one-off check that fails if any commit in the PR contains:

- `Co-Authored-By: Claude` (any variant)
- `Generated by Claude Code`
- `noreply@anthropic.com`

This is a project policy enforcement, not a quality gate. Implementation lives in `scripts/check-forbidden-trailers.sh` in the host repo.
